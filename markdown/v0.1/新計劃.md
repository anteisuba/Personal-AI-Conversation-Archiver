# 一、先給結論：新的 v0.1 方向

這次調整後，我建議你的項目定成：

> **一個 local-first 的 CLI 工具**  
> 將多來源 AI 對話導入、標準化、用 2 次 LLM 整理、寫入 Obsidian，並用 SQLite 管理索引與任務狀態。

---

# 二、根據 Claude 建議後的核心修正

## 1. 包結構收斂

原本 7 個 packages 太細。  
v0.1 改成 **4 個 package** 就夠：

- `apps/cli`
    
- `packages/core`
    
- `packages/ai`
    
- `packages/storage`
    

其中：

- `core`：包含 domain、adapters、shared utils、pipeline orchestration
    
- `storage`：包含 file store、SQLite、markdown exporter
    
- `prompts` 不獨立，先放到 `packages/ai/prompts`
    
- `adapters` 不獨立，先放到 `packages/core/adapters`
    

這樣前期成本低很多。

---

## 2. AI 任務從 6 次 call 壓到 2 次

這個很重要，否則 v0.1 很容易又慢又貴。

### 非 LLM 處理

- 語言識別：本地 heuristic
    
- message_count / token_estimate：本地計算
    
- hash：本地計算
    

### 第 1 次 LLM call

做：

- 分類
    
- 標籤提取
    
- 項目識別
    

### 第 2 次 LLM call

做：

- 摘要
    
- 重寫成知識筆記
    

所以每條對話大致是：

normalize -> analyze_meta -> refine_note

而不是拆成很多小 call。

---

## 3. SQLite 從 v0.1 就支持續跑

這點我同意 Claude。

v0.1 不能只做 happy path，因為你之後很可能一次導入很多對話。  
所以一開始就要做：

- job status
    
- conversation status
    
- retryable pipeline
    
- 斷點續跑
    

至少要做到：

- 哪條已 normalize
    
- 哪條已完成 meta analysis
    
- 哪條已完成 refined note
    
- 哪條 export 成功
    
- 哪條失敗，錯在哪一步
    

---

## 4. data 路徑外置

這個也很重要。

不要把數據綁在 repo 內。  
推薦：

- repo：只存代碼
    
- app data：獨立目錄
    
- vault：獨立目錄
    

例如：

repo: ~/workspace/ai-convo-archiver  
data: ~/.ai-archive/data  
vault: ~/Documents/Obsidian/AI-Vault

這樣：

- git 乾淨
    
- 數據好備份
    
- vault 不耦合 repo
    

---

## 5. CLI 補一個 `run`

這個非常合理。

保留 debug 命令：

- `import`
    
- `process`
    
- `publish`
    

但日常使用時，主入口應該是：

- `run`
    

例如：

ai-archive run ./input/chatgpt-export.json --source chatgpt

這樣符合真實使用習慣。

---

## 6. 分類體系更輕

前期不要把二級分類定太死。

v0.1 只固定一級分類，二級用 tags 動態生成。

### 建議一級分類

- programming
    
- project
    
- career
    
- language
    
- ai-tools
    
- life
    

其他細分先不定死。

---

# 三、調整後的 v0.1 產品目標

我建議你把 v0.1 的範圍明確寫成這樣：

## v0.1 必須完成

- 導入原始對話文件
    
- 標準化為統一模型
    
- 進行本地語言判斷和 token/message 統計
    
- 進行 2 次 LLM 分析
    
- 生成 RefinedNote
    
- 寫入 SQLite
    
- 輸出 Markdown 到 Obsidian
    
- 支持斷點續跑與失敗重試
    

## v0.1 不做

- 實時同步
    
- 瀏覽器插件
    
- Web UI
    
- embedding / vector DB
    
- 相似度聚類
    
- 自動多輪知識合併

## 2. 架構原則

### 原則一：local-first

數據先落本地，不依賴遠端服務保存狀態。

### 原則二：source-agnostic

任何平台導入後都進入統一模型。

### 原則三：Markdown-first

最終知識輸出以 Markdown 為主，不以 DB 為主。

### 原則四：SQLite only for index/state

SQLite 管索引、狀態、任務，不承擔最終知識存儲。

### 原則五：resume-friendly pipeline

任何步驟失敗都能從中斷點繼續。

# 六、推薦的處理流程

新的 v0.1 pipeline 建議如下：

## Step 1: import

輸入原始文件，存入 raw store。

## Step 2: normalize

解析 source 文件，轉成 `NormalizedConversation`。

## Step 3: enrich_local

本地做：

- language detect
    
- message_count
    
- token_estimate
    
- hash
    

## Step 4: analyze_meta

LLM call #1：

- category
    
- tags
    
- project detection
    

## Step 5: refine_note

LLM call #2：

- summary
    
- key points
    
- rewritten note
    

## Step 6: persist

把處理結果寫入：

- processed JSON
    
- SQLite
    

## Step 7: publish

輸出 Markdown 到 Obsidian Vault。

# 七、新的核心數據模型

## 1. NormalizedConversation

export type MessageRole = 'system' | 'user' | 'assistant' | 'tool'  
  
export type Message = {  
  id: string  
  role: MessageRole  
  content: string  
  createdAt?: string  
}  
  
export type NormalizedConversation = {  
  id: string  
  source: 'chatgpt' | 'claude' | 'gemini' | 'cursor' | 'manual'  
  externalId?: string  
  title: string  
  createdAt: string  
  updatedAt?: string  
  messages: Message[]  
  metadata: Record<string, unknown>  
  
  language?: 'zh' | 'ja' | 'en' | 'mixed'  
  messageCount?: number  
  tokenEstimate?: number  
  contentHash?: string  
}

---

## 2. ConversationMetaAnalysis

export type ConversationMetaAnalysis = {  
  category: 'programming' | 'project' | 'career' | 'language' | 'ai-tools' | 'life'  
  tags: string[]  
  project: string | null  
  confidence: number  
}

---

  refinedAt: string

## 4. PublishedNote

export type PublishedNote = {  
  conversationId: string  
  markdownPath: string  
  publishedAt: string  
}
